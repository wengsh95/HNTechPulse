# Comment Stance Classifier

This note records the June 2026 work on improving the `支持 / 质疑 / 中立`
bar chart for Hacker News comment discussion cards.

## Goal

The pipeline already asks the comment judge LLM for `stance_distribution`, but
that value is based on a small selected comment set and is often missing or too
coarse. The local stance classifier is meant to provide a reusable CPU-side
estimate over all fetched comments:

```text
all fetched comments -> local classifier probabilities -> weighted story distribution
```

The output is written beside the normal date artifacts:

```text
data/{date}/stance_distribution.local.json
```

This does not change `comment_judgement.json` or its schema. Render/script code
can opt into the local file and fall back to `comment_judgement.json` when it is
missing.

## Label Definition

The labels are English internally and mapped to Chinese for display:

```text
support -> 支持
skeptic -> 质疑
neutral -> 中立
```

Important labeling rule: stance is judged against the story's core claim, not
against comment sentiment or politeness.

- `support`: the comment strengthens, accepts, or endorses the story's core
  claim, product, technique, result, or direction.
- `skeptic`: the comment weakens, rejects, doubts, or warns against the story's
  core claim, product, technique, result, or direction.
- `neutral`: the comment mainly adds context, implementation detail, history,
  correction, comparison, a question, or mixed tradeoffs without a clear
  support/skeptic stance.

Protect `neutral`: Hacker News has many useful comments that explain boundaries
or correct details without clearly supporting or rejecting the story.

```text

```

The first 360-label attempt was too narrow: labels came from only two stories
(`48398055` and `48403588`). That produced misleading training scores and weak
cross-story behavior. The label sampler was changed to round-robin across
stories before adding more labels.

## Debug Findings

### 1. Story-first input broke MiniLM

The original classifier input was:

```text
[STORY]
long story context

[COMMENT]
comment text
```

`sentence-transformers/all-MiniLM-L6-v2` has `max_seq_length = 256`. With
story-first input, the comment was often truncated away. This caused the first
MiniLM training run to collapse:

```text
accuracy: 0.35
neutral recall: 0.00
```

Fix: sentence-transformers input is now comment-first:

```text
[COMMENT]
comment text

[STORY]
short story context
```

The helper is `embedding_ready_text()` in
`src/pipeline/comment/stance_classifier.py`.

### 2. LogisticRegression regularization was too strong

With 384-dimensional embeddings and a small label set, the default
`LogisticRegression(C=1)` underfit. For the embedding backend,
`C=100` was much better.

### 3. More comments are less useful than more stories

The task is story-relative. The same comment phrase can mean different stances
depending on the story. Example:

```text
This is not new.
```

That can be `skeptic` for a novelty claim, but `neutral` for a historical
discussion. Adding many comments from the same story mostly teaches local
wording. Adding more stories teaches the actual story-relative task.

## Current Evaluation

With 1068 labels and 18 stories, full training-set evaluation for the MiniLM
classifier was:

```text
accuracy: 0.80

support f1: 0.78
skeptic f1: 0.81
neutral f1: 0.80
```

Leave-one-story-out validation is much lower:

```text
MiniLM overall accuracy: 0.419
support f1: 0.321
skeptic f1: 0.483
neutral f1: 0.420

TF-IDF overall accuracy: 0.372
support f1: 0.453
skeptic f1: 0.222
neutral f1: 0.353
```

Interpretation:

- MiniLM is currently better balanced than TF-IDF, especially for
  `skeptic`/`neutral`.
- Single-comment cross-story classification remains hard.
- The bar chart should be evaluated at the story distribution level, not only by
  per-comment argmax accuracy.

## Recommended Next Steps

1. Continue expanding labels by story, not just by comment count.

   Good next target:

   ```text
   50 stories / 1500-2500 labels
   100 stories / 3000-5000 labels
   ```

   Prefer 20-40 labels per story over hundreds from one story.

2. Keep round-robin or bucketed sampling.

   Useful buckets per story:

   ```text
   high-quality comments
   skeptical/risk comments
   support/usefulness comments
   experience/production comments
   correction/history/comparison/question comments
   ordinary top-level comments
   ```

3. Add story-level distribution evaluation.

   The product need is the bar chart, so evaluate:

   ```text
   local story distribution vs LLM story-level distribution
   ```

   Do not rely only on single-comment accuracy.

4. Consider probability calibration.

   The chart uses summed probabilities, so calibration matters. Candidate
   approaches:

   ```text
   CalibratedClassifierCV
   temperature scaling
   story-level smoothing with comment_judgement.stance_distribution
   ```

5. Wire rendering carefully.

   Recommended behavior:

   ```text
   if data/{date}/stance_distribution.local.json exists:
       use local stance_distribution for the bar chart
   else:
       use comment_judgement.stance_distribution
   ```

   This keeps existing artifacts compatible and allows local stance reports to
   be generated or regenerated independently.
