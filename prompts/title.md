## 标题生成规则

【标题目标】
生成类似以下风格的标题（注意：不要使用"社区XX"句式）：
- 《【HN观察】Claude Fable 5 发布：技术突破还是 IPO 造势？》
- 《【HN观察】Claude 被曝"静默降智"：付费用户不能被暗中削弱》
- 《【HN观察】微软 70 个 AI 仓库被投毒：这锅别甩给开源》
- 《【HN观察】Claude 新模型更安全了：开发者却担心它更难用了》
- 《【HN观察】Claude 降智争议升级：安全机制变成商业武器》

【强制要求】
1. 每个标题必须以 **【HN观察】** 开头。
2. 每个标题结构必须是 **【HN观察】事件描述：态度/争议焦点**（冒号后表达态度或争议）。
3. 禁止使用"社区质疑""社区吵翻""社区警觉"等"社区XX"句式。态度直接表达，例如用"付费用户不能被暗中削弱"而不是"社区怒批：付费用户不能被暗中削弱"。
4. 禁止只写事件，不写态度/争议。
5. 禁止只写态度，不写具体事件。
6. 禁止虚构评论中没有出现的立场。
7. 如果评论以质疑为主，态度部分用"质疑""不满""担心""警觉"等词直接表达。
8. 如果评论分歧明显，态度部分用"争议""分歧""两极分化"等词直接表达。
9. 标题要适合 B 站科技区：有冲突、口语化、不标题党、不过度煽动。
10. **标题长度严格控制在 18-40 个字符（中文字、英文字母、标点、空格都算 1 个字符，包含【HN观察】前缀）**。生成前必须数一遍每个标题的字符数，超过 40 的必须缩短。
11. **必须恰好输出 3 个候选标题**，按点击潜力从高到低排序。少一个都不行。

【推荐标题结构】
A. 《【HN观察】事件描述：争议焦点？》
B. 《【HN观察】事件曝光/发布/翻车：核心矛盾》
C. 《【HN观察】事件引发争议：潜在后果》
D. 《【HN观察】事件升级：不能……》

【品牌与主语规则】
- 如果主事件里有强品牌、平台或责任主体，必须把它放到标题开头，例如 `NVIDIA`、`Anthropic`、`微软`、`Steam`。不要把泛机制词抢主语。
- 标题结构优先是"品牌/平台 + 做法/变化 + 后果"，例如 `Meta 用 AI 客服，账号被盗`。
- 安全事故标题：受害者/平台 + 出事动词 + 数字/具体后果。动词必须准确：`账号被盗`、`账号被接管` 优于 `账号交出去`。

【冲突词清单】
- 事实明确时可用：掉线、涨价、封禁、停服、被盗、被接管、失守、被滥用、关闭、砍掉。
- 事实不明确时降级：延迟升高、绕路、被质疑、引发争议、用户反馈。
- 禁止：震惊、炸裂、重磅、全网、彻底改变、时代变了、封神、翻车了。

【动词准确性】
- 安全事故：被盗 / 被劫 / 被接管 / 失守 / 被滥用，禁止"交出账号 / 放走账号 / 送走账号"。
- 归因有争议时用"被质疑 / 引争议 / 绕路 / 延迟飙升 / 用户反馈"，不要替争议下结论。

【其他限制】
- 不编造数字、年份、公司动作、法律结果、用户规模；所有具体数字必须来自输入。
- 不用 emoji、感叹号、问号、方括号装饰、营销口号。
- 中文优先，必要英文产品名保留；中英文混排紧凑。
- 主推标题只押一个主冲突，不要两个 story 硬拼。
- 3 个候选角度要不同：后果型、机制型、反常识型等。

## title

- `title_candidates` **必须恰好 3 个**，按点击潜力降序；第一候选同步写入 `title`。少一个或多一个都不行。
- 每个候选标题 18-40 个字符（中英文、标点、空格各算 1 字符，包含【HN观察】前缀）。生成后必须逐个数一遍，超长的删字缩短。
- 每个标题必须以 **【HN观察】** 开头，结构为 **【HN观察】事件：态度/争议**。
- 禁止使用"社区质疑""社区吵翻""社区警觉"等"社区XX"句式。
- 主推标题必须能回答"点开后我会看到什么冲突"。如果只是"X 与 Y 发生了"，要改成"X 做了什么，导致/结果谁付出什么代价"。
- 候选之间不能只是替换同义词；必须分别测试不同点击心理：具体受害者、行业代价、反常识事实。
- 如果出现两个互不相关的公司/产品/事件名在同一个标题中，重写成单一主事件。
- 不要使用"今日速览""AI资讯""科技早报""HN每日观察"作为标题主体。

## description

- 2-3 句话，简洁明了。
- 第一句先展开标题承诺的主事件，给出机制、数字、后果或受影响人群。如果标题为了点击做了白话因果压缩，第一句必须补足真实链路。
- **使用 `article_summary` 和 `key_points` 中的具体细节**（技术机制、数字、影响范围、时间线等），不要泛泛而谈。例如：
  - 如果 `key_points` 提到"定价翻倍"，description 要写出来
  - 如果 `article_summary` 提到"5% 请求回退 Opus 4.8"，description 要体现
  - 如果有具体数字（Token 价格、日期、用户数），必须用上
- 如果 `key_points` 提供了"为什么关注"或"影响"等信息，description 应该体现这些内容。
- 最后一句落到成本/风险/控制权由谁承担，不要泛泛升华。
- 不要复读标题；不要写"本期视频将带你了解"。
- 不要写"你怎么看，欢迎在评论区聊聊"这种套话。

## cover_title / cover_tags

- `cover_title` 是封面缩略图上最大的一行字，12-22 个中文字符，必须比 `title` 更短。
- 优先用短白话因果句或强结果句：`Meta 用 AI 客服，账号被盗`。
- 如果主推标题里有数字后果，`cover_title` 必须保留数字。
- `cover_title` 如果涉及强品牌，必须保留品牌开头或第一视觉位置。
- `cover_tags` 输出 1-2 个短标签，每个 4-10 个中文字符；用于补充主事件的角度，不重复 `cover_title` 的主信息。

## cover_subtitle

- 三行，每行以 `— ` 开头。
- 每行 6-12 个中文字符或等长中英混排短语。只写短语，不写完整句，不加句号。
- 三行都服务封面主事件：第一行写具体对象，第二行写冲突/代价，第三行写机制/受影响人群。

## tags

- 5-8 个中文标签。主题优先，具体公司/产品不超过 2 个。
- 适合 B 站科技区检索，例如：`AI`、`开发者工具`、`开源`、`游戏产业`、`网络协议`。

## 失败保护

- 如果输入不足以生成安全文案，输出空 `title / description / tags`。
- 如果无法判断某个高冲突词是否被事实支持，降级为中性词：`影响`、`绕路`、`延迟升高`、`被迫改用`、`引发争议`、`用户反馈`。
