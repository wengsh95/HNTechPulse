## 任务

为一条 Hacker News story 生成产品化的三段叙事短视频脚本。每条 story 必须回答：

1. 发生了什么，以及为什么值得技术读者看一眼。
2. 社区怎么讨论：情绪、分歧、争议焦点或补充细节。
3. 哪个代表观点能让这条新闻留下记忆点。

画面使用三张卡片承载：事件卡 -> 氛围卡 -> 观点卡。

语言要口语化、有信息密度，所有上屏字段必须短而清楚。旁白必须音频自洽：观众就算不看画面，只听声音，也能理解发生了什么。

---

## 预提取字段

输入 `story_json` 中已包含预提取的 EventCard 字段（`editor_angle`、`dek`、`key_points`、`keywords`、`category`、`visual_hint`），由上游 enrichment 步骤生成。这些字段已确定，**不要重新生成或改写**，直接作为上下文使用。

- 用 `editor_angle` 和 `dek` 理解事件核心，据此写口播旁白。
- 用 `key_points` 确认背景和影响，旁白中可口语化转述但不要逐字重复。
- 用 `category` 选择叙述模式。
- 用 `keywords` 辅助理解主题。

## 三段旁白写法

三段 `audio_text` 拼起来必须像一段自然口播，而不是三段摘要。每段只完成一个任务，但三段合起来必须独立讲明白这条新闻。

### event_card

作用：用 1-2 句话讲清楚"发生了什么"。

写法：

- 第一小句必须明确说出主语、动作和对象，不要让观众依赖画面标题补全信息。
- 第二小句解释它为什么值得技术圈看一眼。
- 不要使用"引发关注""备受争议""值得关注"这类空话。
- 不要把评论区观点提前写进这一段。

长度：45-70 个中文字符。

### atmosphere_card

作用：讲清楚评论区真正的分歧、补充或吐槽。

写法：

- 不要固定写"社区两极分化"。
- 必须指出具体分歧对象，比如"厂商控制权 vs 用户所有权""安全执法 vs 加密完整性"。
- 如果评论区不是强对立，就写成"大家主要在补充细节、纠错、怀旧或吐槽实现"。
- 不要为了制造冲突而夸大争议。
- 不要重复 event_card 已经说过的事件背景。

长度：45-65 个中文字符。

### quote_card

作用：用评论视角给这条新闻一个记忆点。

写法：

- 不要堆两三个长引用。
- 优先转述观点，不要大段直引。
- 只保留一个最有代表性的评论态度或反问。
- 结尾要落在具体问题上，不要升华成宏大价值判断。
- 不要重复 atmosphere_card 已经说过的分歧。

长度：50-80 个中文字符。

## 叙述模式

根据 story 内容选择一种口播模式，不要所有条目都写成同一种争议摘要。

可选模式：

1. 新闻型：适合政策、公司动作、安全事件。重点是"发生了什么 + 影响谁"。
2. 技术型：适合工具、语言、库、基础设施。重点是"解决什么问题 + 代价是什么"。
3. 社区型：适合 Show HN、个人项目、怀旧项目。重点是"为什么大家觉得有趣 + 评论区怎么吐槽"。
4. 风险型：适合隐私、加密、供应链、安全。重点是"风险边界 + 反对意见"。
5. 反常识型：适合标题有冲突或荒诞感的故事。重点是"看起来奇怪，但它暴露了什么问题"。

生成时只使用对应模式的语言，不要每条都写"社区两极分化"。

## 中文口播质量要求

这不是新闻稿，也不是文章摘要。它是一段中文技术短视频口播。

观众是懂一点技术的中文开发者，但不一定熟悉每个具体领域。

`audio_text` 必须像真人说话，不要像公司新闻稿、政策通报或论文摘要。优先用短句、动词和具体对象，少用抽象名词。

旁白按"音频优先"写。观众就算把手机放在一边，只听声音，也必须能听懂：

1. 谁或哪个项目做了什么。
2. 这件事影响什么人或什么系统。
3. 评论区在争什么、补充什么或担心什么。

三段 `audio_text` 拼起来必须像一个人在自然讲述：

1. 先讲发生了什么。
2. 再讲评论区具体吵什么、补充什么或吐槽什么。
3. 最后留下一个有代表性的观点或反问。

不要把关键信息只放在画面里。`audio_text` 里必须出现足够的主语和上下文。

每段只允许有一个主信息点：

- `event_card`：只讲事件，不讲评论区。
- `atmosphere_card`：只讲分歧、补充或吐槽，不重复事件。
- `quote_card`：只讲一个代表观点，不再补充背景。

如果一个事实已经出现在预提取的 `dek` 或 `key_points`，不要在三段旁白里逐字重复；但如果它是理解事件所必需的主语、动作或对象，必须用更口语的方式在 `audio_text` 中重新交代。

如果一个观点已经出现在 `atmosphere_card`，不要在 `quote_card` 里换一种说法重复。

不要用"它""这件事""这个项目"开头，除非上一句已经明确说出所指对象。

第一次出现英文项目名、公司名或技术名时，如果普通听众可能不熟悉，必须加一个极短解释。例如："OrcaSlicer 这个 3D 打印切片软件""Zed 这个代码编辑器"。

## 口语化硬约束

`audio_text` 要像一个懂技术的人在跟朋友讲，不要像播新闻。

优先使用：

- "现在"
- "这次"
- "说白了"
- "麻烦的是"
- "评论区吵的是"
- "有人担心"
- "有人补了一刀"
- "听起来小，但影响不小"
- "这点很 HN"

避免使用书面词和播报腔：

- 正式宣布
- 正式支持
- 正式强制要求
- 暴露了
- 揭示了
- 标志着
- 反映了
- 推动了
- 赋能
- 基线
- 生态
- 路径
- 程序本身
- 价值所在
- 伦理
- 痛点
- 代价
- 普及的原因
- 理想照进现实

把书面表达改成人话：

- 错误：Debian 发行团队正式强制要求所有软件包必须可重现构建。
- 正确：Debian 现在要求软件包必须能重复构建，换台机器编出来也要一样。

- 错误：这大幅提升供应链安全基线。
- 正确：这样别人才能核对软件包有没有被偷偷改过。

- 错误：这暴露了大厂强制 AI 转型的内部代价。
- 正确：公司想拿更多训练数据，员工却觉得自己的工作电脑没边界了。

- 错误：它揭示了这个以安全著称的国家可能存在的制度风险。
- 正确：文章提醒你，到了国外，法律流程本身也可能变成风险。

- 错误：这是 JVM Clojure 没能在 serverless 和 CLI 领域普及的原因。
- 正确：很多人不用 Clojure 写小脚本，就是嫌 JVM 启动太慢。

每段 `audio_text` 至少有一句要能直接念给朋友听，不需要改写就自然。

## 长度和节奏硬约束

每条 story 的三段 `audio_text` 总字数必须控制在 160-210 个中文字符之间。优先保证音频自洽，再控制长度。

- `event_card`：50-70 字，最多 2 句。必须交代主语、动作、对象。
- `atmosphere_card`：45-60 字，最多 2 句。必须交代具体分歧对象。
- `quote_card`：45-70 字，最多 2 句。必须能独立听懂评论在回应什么。

如果信息太多，优先删除：

1. 宏大评价。
2. 背景铺垫。
3. 第二个评论观点。
4. 形容词和副词。

禁止每段写成 3 句以上。

禁止在同一段里同时讲背景、争议和评论。

禁止用长句硬塞信息；宁可少说一个细节，也不要破坏口播节奏。

## 事实确定性规则

必须区分事实、报道、作者观点、评论区推测。

- 如果输入只来自文章或评论，不要写成官方宣布。
- 如果是提案，用"提议""建议""正在推动"，不要写"已经实施"。
- 如果是报道，用"报道称""文章称"。
- 如果是评论观点，用"有评论认为""评论区担心"。
- 不要补充输入里没有的动机、规模、后果。
- 不要为了让故事更强而扩大影响范围。
- 数字、规模、金额、日期、百分比必须来自输入材料。
- 如果输入没有明确数字，不要编数字，也不要把模糊范围写成精确数字。
- 如果是作者个人经历，用"作者称""文章记录"，不要写成普遍事实。
- 涉及法律、国家制度、司法、公共政策时，必须使用"文章称""报告建议""评论区质疑"等来源标记，不要做节目方定性。

错误：Meta 审布全面监控员工。

正确：报道称 Meta 正扩大员工操作数据采集，用于 AI 训练。

错误：法国将打破所有加密通信。

正确：法国议会相关报告建议，为加密通信引入定向访问机制。

错误：日本拘留制度会造成永久后果。

正确：文章称，长时间失联可能让当事人面临工作和签证风险。

## 评论卡写法

`quote_card` 不是评论摘抄集合，而是"评论区记忆点"。

优先级：

1. 一个最有洞察的观点。
2. 一个最能代表分歧的反问。
3. 一个轻微幽默但不低俗的吐槽。

不要同时塞入三种声音。

不要使用"一位用户说 / 另一位用户说 / 还有人说"的流水账结构。

优先不要使用"最高赞""高赞评论"。除非评论排序信息对理解很重要。

不要为了对称而强行写支持方和反对方。

推荐句式：

- 有人担心的点其实很具体：……
- 有人把问题说得很直：……
- 评论区最有 HN 味的一点是：……
- 反对者并不是反对目标，而是怀疑这个实现方式。

## 风格示例

差：

社区情绪分裂：一方支持维修权，批评厂商封锁行为；另一方质疑分支是否过度连接私有云 API，认为法律威胁有据。争议背后，是维修权与厂商控制权之间的残酷拉扯。

好：

评论区真正吵的是边界：维修权能不能覆盖对私有云 API 的绕行。支持者看的是用户所有权，反对者担心这是在冒充官方客户端。

差：

这一事件引发开发者社区广泛关注，也让人重新思考开源生态和商业公司的关系。

好：

这事麻烦的地方在于，两边都能讲出道理：开源社区讨厌封锁，厂商也不想私有接口被第三方客户端拿来直连。

差：

争议背后，是技术野心与人的价值之间的残酷拉扯。

好：

说白了，公司想要更多训练数据，员工想知道自己的工作电脑到底还有没有边界。

## 其他字段说明

- `debate_focus`：2-3 个争议焦点短语，概括社区在争论哪几个核心问题。写在顶层 JSON 和 `atmosphere_card.props` 里。如果没有强争议，写"实现细节""历史背景""适用场景"等具体讨论焦点。
- `card_narrations`：三段旁白文案，按 event_card -> atmosphere_card -> quote_card 顺序。视频中会按此顺序展示对应卡片，每张卡片的显示时长等于对应 `audio_text` 的 TTS 时长。
- `stance_distribution`：社区态度分布估计，3-5 个 stance，value 为 0-1，占比总和必须为 1。stance 使用：支持、质疑、中立、调侃、担忧。
- `quote_card` 必须包含 story_index；在 `selected_comment_ids` 里优先填满 3 个输入 `comments[].id`，分别代表不同角度的有态度评论。确实不足 3 条合格评论时才少于 3 个；没有合格评论时输出空数组。

## quote_card 选评论硬约束

`quote_card.props.selected_comment_ids` 只能选择输入 `comments[].id` 里的 id。默认必须选择 3 条，且三条最好分别覆盖不同角度，例如支持、质疑、经验补充、技术取舍、风险担忧、反问或吐槽。选中的评论必须有明确观点、态度、经验、反问、技术取舍或具体担忧。

以下评论不能进入 `selected_comment_ids`：
- 主要内容是链接、论文、文档、博客或资料推荐。
- 去掉 URL 后只剩"这里有一篇文章""相关资料""可以看看"这类话。
- 没有表达支持、反对、担心、经验、怀疑、反问或具体技术代价。
- 只是复述标题、表示感谢，或需要父评论上下文才能理解。

资料型评论可以帮助你理解背景，但不能当作代表观点上屏。如果没有合格评论，`selected_comment_ids` 输出 `[]`，`quote_card` 旁白改为转述评论区整体观点，不要硬选链接评论。

## Constraints

- 只输出 JSON。
- 翻译准确自然，技术术语保留英文或给标准译名。
- 尊重事实，不做想当然的修改。
- `card_narrations` 中的 `audio_text` 拼接后应自然流畅，适合口语播报。
- 不要只复述标题；每条 story 必须传递具体信息。
- 避免空话，必须写出具体对象、风险、代价、边界或趋势。
- 不要为了制造戏剧性而夸大争议。
- 不要输出 Markdown，不要输出代码块，只输出 JSON 对象。

## 合规性

- 如果评论或文章内容涉及政治敏感、暴力、色情、仇恨言论、个人隐私泄露等不合规内容，忽略该内容，不要引用、转述或摘要。
- 不要对不合规内容做任何形式的传播，包括反讽、调侃或"打码"引用。
- `audio_text` 中不得出现任何可能触发内容审核的表述，保持中性客观。
