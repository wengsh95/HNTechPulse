## 任务

为一条 Hacker News story 生成三段叙事短视频脚本。画面为三张卡片：事件卡 → 氛围卡 → 观点卡。

三段旁白拼起来必须像一个人自然讲述：发生了什么 → 评论区在吵什么/补充什么 → 一个有记忆点的评论视角。

语言：口语化、有信息密度、音频自洽（不看画面只听声音也能懂）。

---

## 预提取字段

输入 `story_json` 包含上游 enrichment 已生成的字段，**不要重新生成或修改**，直接作为上下文：

- `editor_angle` / `dek`：理解事件核心，据此写旁白。
- `key_points`：确认背景和影响，可口语化转述但不要逐字重复。
- `category`：选择叙述模式。
- `keywords`：辅助理解主题。

## 各卡片写法

### EventCard

**`subtitle_texts[0]`：一句话讲清谁做了什么。**
**`subtitle_texts[1]`：说为什么值得看。**

- 第一句必须有主语、动作、对象，不要让观众依赖画面标题补信息。
- 不要用"引发关注""备受争议""值得关注"这类空话。
- 不要把评论区观点提前写进来。
- 段尾不要落宏大评价。

每条 20-35 字，各自独立成句。

### AtmosphereCard

**`subtitle_texts[0]`：评论区在吵什么或补什么。**
**`subtitle_texts[1]`：分歧根源或最犀利的补刀。**

- 必须指出具体分歧对象，如"厂商控制权 vs 用户所有权""安全执法 vs 加密完整性"。
- 如果不是强对立，就写"大家主要在补充细节、纠错、怀旧或吐槽实现"，不要硬造争议。
- 不要重复 EventCard 说过的事件背景。

每条 20-30 字，各自独立成句。

### QuoteCard

**`subtitle_texts[0]`：转述一个评论态度或反问，给这条新闻一个记忆点。**
**`subtitle_texts[1]`：落在具体问题上，不要升华。**

- 基于系统预选的评论候选（见输入区 `quote_candidates`）来写，不要再自行筛选。
- 只保留一个最有代表性的评论态度或反问。
- 结尾落在具体问题上，不要升华成宏大价值判断。
- 不要重复 AtmosphereCard 说过的分歧。
- 推荐句式："有人担心的点其实很具体：…""有人把问题说得很直：…""评论区最有 HN 味的一点是：…"

每条 20-30 字，各自独立成句。

## 叙述模式

根据内容选择一种口播语调，不要所有条目都写成同一种争议摘要：

| 模式 | 适用 | 重点 |
|------|------|------|
| 新闻型 | 政策、公司动作、安全事件 | 发生了什么 + 影响谁 |
| 技术型 | 工具、语言、库、基础设施 | 解决什么问题 + 代价是什么 |
| 社区型 | Show HN、个人项目、怀旧项目 | 为什么有趣 + 评论区怎么吐槽 |
| 风险型 | 隐私、加密、供应链、安全 | 风险边界 + 反对意见 |
| 反常识型 | 标题有冲突或荒诞感 | 看起来奇怪，但暴露了什么问题 |

## 写作规范

### 口语化：像懂技术的人跟朋友讲，不要像公司新闻稿

**多用：** "现在""这次""说白了""麻烦的是""评论区吵的是""有人担心""有人补了一刀""听起来小，但影响不小""这点很 HN"

**禁用以下书面词和播报腔：** 正式宣布/支持/强制要求、引发关注/热议/广泛讨论、暴露了、揭示了、标志着、反映了、推动了、赋能、基线、生态、路径、价值所在、伦理、痛点、理想照进现实

**禁用以下情绪和套话词：** 彻底、非常愤怒、没底线、不惜一切、杀伤力、荒谬、无情、永久后果、灾难、毁灭、崩塌、重创、震撼；社区两极分化/情绪分裂、炸锅、风暴、重磅、残酷拉扯、值得深思、时代信号、这背后是、一场关于

**正确 vs 错误：**

> ✗ Debian 发行团队正式强制要求所有软件包必须可重现构建。这大幅提升供应链安全基线。
> ✓ Debian 现在要求软件包必须能重复构建，换台机器编出来也要一样。这样别人才能核对软件包有没有被偷偷改过。

> ✗ 争议背后，是技术野心与人的价值之间的残酷拉扯。
> ✓ 说白了，公司想要更多训练数据，员工想知道自己的工作电脑到底还有没有边界。

> ✗ 社区情绪分裂：一方支持维修权，批评厂商封锁行为；另一方质疑分支是否过度连接私有云 API。争议背后，是维修权与厂商控制权之间的残酷拉扯。
> ✓ 评论区真正吵的是边界：维修权能不能覆盖对私有云 API 的绕行。支持者看的是用户所有权，反对者担心这是在冒充官方客户端。

### 克制情绪：批评写具体后果，不写情绪判断

> ✗ 这比任何反驳都更具杀伤力。
> ✓ 这个评论点出了更现实的问题：员工信任一旦丢了，监控可能反而制造更多泄露风险。

### 事实准确：区分事实、报道、评论、推测

- 报道标"报道称""文章称"；评论标"有评论认为""评论区担心"；提案写"提议/正在推动"而非"已经实施"。
- 不补充输入里没有的动机、规模、后果。数字/日期/百分比必须来自输入材料。
- 涉及法律、司法、公共政策时，必须使用来源标记，不做节目方定性。

> ✗ 法国将打破所有加密通信。
> ✓ 法国议会相关报告建议，为加密通信引入定向访问机制。

### 长度控制

所有 `subtitle_texts` 总字数 130-180 字符（约 6 条，每条 20-35 字）。信息太多时优先删除：宏大评价 > 背景铺垫 > 第二个评论观点 > 形容词/副词。

每条独立成句。禁止用长句硬塞信息。禁止一条字幕同时讲两件事。

## 其他字段

- `emotion`：根据故事整体情绪选择 neutral / upbeat / concerned / curious。
- `quote_angle`：基于系统预选的评论候选，提炼一个核心观点，12-28 个中文字符。评论的最终选取以及立场分布、争议焦点均由系统完成，你不需要输出这些字段。

---

## 合规性

- 涉及政治敏感、暴力、色情、仇恨言论、个人隐私泄露等内容，忽略不引用。
- 不对不合规内容做任何形式的传播，包括反讽、调侃或"打码"引用。
- `audio_text` 中不得出现可能触发内容审核的表述。
