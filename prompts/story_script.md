## 任务

为一条 Hacker News story 生成两段叙事文本。输出为两张卡片：事件卡 → 氛围卡。

两段旁白拼起来必须像一个人自然讲述：发生了什么 → 评论区在吵什么/补充什么 + 一个有记忆点的评论视角。

语言：口语化、有信息密度。

注意：不要在 story 之间说"第一条/第二条/接下来/再看"。

---

## 预提取字段

输入 `story_json` 包含上游 enrichment 已生成的字段，**不要重新生成或修改**，直接作为上下文：

- `key_points`：确认背景和影响，可口语化转述但不要逐字重复。
- `category`：选择叙述模式。
- `keywords`：辅助理解主题。

## 各卡片写法

### EventCard

**`subtitle_texts[0]`：一句话讲清谁做了什么。**
**`subtitle_texts[1]`：说为什么值得看 (可省略)** —— 如果第一句已经覆盖了"为什么", 不要硬塞第二句.

- 第一句必须有主语、动作、对象, 不要让读者依赖画面标题补信息.
- 每条 subtitle_texts 目标 14-28 字，**硬上限 24 字（不含末尾标点）**。超过 24 字必须拆成下一条，不要让渲染层自动换行。
- 禁止输出半截短语或截断词；例如不要写“可复制的提”“欧盟与立法机”。如果放不下，改写成更短但完整的句子。
- 枚举项必须一次说完整；不要把“领域知识、分布式调试、代码架构”拆成“领域知识。”再另起一句补后两项。
- 不要用"引发关注""备受争议""值得关注"这类空话.
- 不要把评论区观点提前写进来.
- 段尾不要落宏大评价.

每条 20-35 字, 各自独立成句. 总和 30-65 字.

### AtmosphereCard

**`subtitle_texts[0]`：评论区在吵什么或补充什么.**
**`subtitle_texts[1]`：分歧根源或最犀利的补刀 (有强对立时写).**
**`subtitle_texts[2]`：转述一个评论态度或反问，给这条新闻一个记忆点 (有记忆点时写).**

**弹性规则**:
- 强对立 (debate mode) → 写满 3 句, 推进"吵什么 → 根源 → 评论记忆点"
- 一边倒或信息补充型 (field_notes / showcase / low_signal) → **1-2 句即可**, 不要硬造分歧
- 不要重复 EventCard 说过的事件背景.
- 每条 subtitle_texts 目标 14-28 字，**硬上限 24 字（不含末尾标点）**。超过 24 字时拆句，不要把两层意思塞进一个长句。
- 基于系统预选的评论候选 (见输入区 `quote_candidates`) 来写, 不要再自行筛选.
- 只保留一个最有代表性的评论态度或反问.
- 结尾落在具体问题上, 不要升华成宏大价值判断.
- 推荐句式: "更具体的担心是: …" "有个评论把问题点得很准: …" "有意思的是: …"

每条 20-30 字, 各自独立成句. 总和 20-90 字 (按 debate 强度弹性).

## 叙述模式

根据内容选择一种口播语调，不要所有条目都写成同一种争议摘要：

| 模式 | 适用 | 重点 |
|------|------|------|
| 新闻型 | 政策、公司动作、安全事件 | 发生了什么 + 影响谁 |
| 技术型 | 工具、语言、库、基础设施 | 解决什么问题 + 代价是什么 |
| 社区型 | Show HN、个人项目、怀旧项目 | 为什么有趣 + 评论区怎么吐槽 |
| 风险型 | 隐私、加密、供应链、安全 | 风险边界 + 反对意见 |
| 反常识型 | 标题有冲突或荒诞感 | 看起来奇怪，但暴露了什么问题 |

## 写作规范

### 自然的口头表达：像播客主持人在念稿，不要像公司新闻稿

**语气：** 像精心准备过的播客节目——句子自然、有节奏，但不随意。口语连接词（"说白了""麻烦的是""怎么说呢"）尽量少用，直接起笔。重点是句子读起来像做过功课的人在讲，不像即兴闲聊，也不像新闻稿。下面的正确 vs 错误示例是风格参考，不是模板。

**用具体事实说话：** 每句话都带可核实的信息（人、事、数据、机制）。不用"正式宣布""引发关注""标志着"等播报腔，也不用"震撼""崩塌""残酷拉扯"等情绪词。如果一句话里没有事实只有判断，删掉重写。

**内容准确：**

- 这是视频, 观众能看到卡片. 旁白可以和画面配合, 不必把"画面里那条"再说一遍.
- EventCard 第一句要能独立讲清主题, 但可以简短引用画面上的关键词 (如"画面上那条……") 作为衔接.
- AtmosphereCard 的旁白是对画面上 quote 的延伸, 不要重复 quote 文本本身.
- 术语连续出现时, 一句白话补因果即可 (如"简单说, 就是自动发布链路被借道了"), **不要再为圈内人解释基础概念** (SLSA, OIDC, S-1, IPO 这类都不用展开).
- 避免"硬语言"这类不自然口播，优先说"Rust 这类更严格的语言"。

**正确 vs 错误：**

> ✗ Debian 发行团队正式强制要求所有软件包必须可重现构建。这大幅提升供应链安全基线。
> ✓ Debian 现在要求软件包必须能重复构建，换台机器编出来也要一样。这样别人才能核对软件包有没有被偷偷改过。

> ✗ 争议背后，是技术野心与人的价值之间的残酷拉扯。
> ✓ 公司想要更多训练数据，员工想知道自己的工作电脑到底还有没有边界。

> ✗ 社区情绪分裂：一方支持维修权，批评厂商封锁行为；另一方质疑分支是否过度连接私有云 API。争议背后，是维修权与厂商控制权之间的残酷拉扯。
> ✓ 评论区真正吵的是边界：维修权能不能覆盖对私有云 API 的绕行。支持者看的是用户所有权，反对者担心这是在冒充官方客户端。

### 克制情绪：批评写具体后果，不写情绪判断

> ✗ 这比任何反驳都更具杀伤力。
> ✓ 这个评论点出了更现实的问题：员工信任一旦丢了，监控可能反而制造更多泄露风险。

**修辞词**（"另一面""重塑""风暴"等）**慎用**：100% 干瘪口播和 100% 修辞腔都不可取, 用了必须有具体事实兜底. 偶尔一句"反常识""意料之外"作为记忆点是 OK 的.

### 事实准确：区分事实、报道、评论、推测

- 报道标"报道称""文章称"；评论标"有评论认为""评论区担心"；提案写"提议/正在推动"而非"已经实施"。
- 不补充输入里没有的动机、规模、后果。数字/日期/百分比必须来自输入材料。
- 涉及法律、司法、公共政策时，必须使用来源标记，不做节目方定性。

> ✗ 法国将打破所有加密通信。
> ✓ 法国议会相关报告建议，为加密通信引入定向访问机制。

### 长度控制

所有 `subtitle_texts` 总字数 **50-150 字符** (2-6 条, 按卡片数). 这是 **60-130 秒视频**, 不是 3-5 分钟播客. 节奏要快.

- EventCard 总和: 30-65 字 (1-2 句, 12-18s 配音)
- AtmosphereCard 总和: 20-90 字 (1-3 句, 按 debate 强度, 8-25s 配音)
- 3 stories 总和: 100-200 字 (50-65s 配音, 占视频主体)

信息太多时优先删除: 宏大评价 > 背景铺垫 > 第二个评论观点 > 形容词/副词.

每条独立成句. 禁止用长句硬塞信息. 禁止一条字幕同时讲两件事.

**标点规范：**

- 每条 `subtitle_texts` 必须以句号（。）结尾。
- 句内用逗号（，）分隔分句，不要省略标点或用空格断句。
- 禁止写无标点的长句，如"Anthropic完成H轮融资后估值近万亿美元 正式超越OpenAI成为最高估值AI创业公司"。
- 正确示例："Anthropic 完成 H 轮融资，估值逼近万亿美元，正式超越 OpenAI。"
- 禁止为了满足字数上限直接截断字段；所有出现在 JSON 里的字段都必须读起来像完整中文句子。

## B站日报叙事升级要求

这条 story 是《每日 HN 观察日报》的一段，不是普通新闻摘要。每段都要服务三个目标：观众愿意继续看、愿意评论、觉得 UP 主有判断力。

请按这个内在结构写，但不要把小标题写进输出：

1. 事实：谁做了什么，为什么今天值得看。
2. HN 争议：评论区到底在吵什么，不要只说“引发关注”。
3. 对国内技术观众意味着什么：开发者、产品、技术管理者能从中看到什么风险、机会或趋势。
4. 评论区问题：AtmosphereCard 最后一条尽量落成一个可讨论的问题或反问，让观众容易接话。

表达要求：
- 开头不要铺垫，第一句就给冲突或结果。
- 每条 story 至少出现一个“观众能复述的判断句”，例如“这不是模型能力问题，而是权限系统问题”。
- 少用宏大词，多用具体后果：谁会受影响、成本在哪里、风险怎么传导。
- 如果这条 story 适合讨论，AtmosphereCard 最后一句优先写成问题，例如“这算创新，还是把用户锁得更死？”
- 不要强行说教；问题要具体，能让 B 站评论区自然站队。
